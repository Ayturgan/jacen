# Regression сценарии для непрерывности кампании

## Цель
Проверить, что новая архитектура (System Prompt + Lore + RAG + State + Memory + Summarizer) не теряет канон и последствия.

## Сценарий 1: Длинная сессия
- Стартовать квест `/quest`.
- Провести 15+ ходов с разными типами действий.
- Проверить, что `scene_memory`, `session_memory`, `campaign_summary` и `campaign_open_loops` обновляются.
- Ожидание: нет мета-утечек, фаза сцены и давление меняются последовательно.

## Сценарий 2: Пауза и возобновление
- Во время активной сцены сделать `/quest pause причина`.
- Попробовать обычный ход игрока.
- Возобновить `/quest resume`.
- Ожидание: до `resume` бот не продвигает сцену; после `resume` продолжает без потери контекста.

## Сценарий 3: Переход акта
- Довести сцену до условий автоперехода.
- Проверить событие `act_auto` и старт следующего акта.
- Ожидание: перед переходом запускается forced summarizer, прошлый акт отражён в памяти.

## Сценарий 4: Канонический конфликт
- Спровоцировать ответ, который пытается ретконить факт (например, «этого не было»).
- Ожидание: guardrails смягчает реткон, в логе есть `guardrail` событие.

## Сценарий 5: RAG наблюдаемость
- Сделать ход с явной привязкой к лору персонажа.
- Проверить, что в `game_log` есть `trace` с источниками `CANON/SCENE/PERSONAL/SEMANTIC`.
- Ожидание: в `trace` указаны источники и размеры memory-снимков.

## Сценарий 6: Paid/Free мозг
- Переключить `/brain free`, затем `/brain paid`.
- Сделать по одному ходу в каждом режиме.
- Ожидание: в `trace` отражается корректный `tier`; paid остаётся Gemini-only.

## Критерии прохождения
- Нет синтаксических ошибок и падений обработчиков.
- Нет противоречий в `campaign_summary` после переходов.
- Открытые нити не теряются и не дублируются бесконечно.
- Команды админа и админка меняют состояние предсказуемо.
